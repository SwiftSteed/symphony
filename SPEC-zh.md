# Symphony 服务规范

状态：草案 v1（语言无关）

目的：定义一种编排编码代理以完成项目工作的服务。

## 规范性用语

本文档中的关键词 MUST（必须）、MUST NOT（绝不允许）、REQUIRED（强制）、SHOULD（应当）、SHOULD NOT（不应当）、RECOMMENDED（推荐）、MAY（可以）以及 OPTIONAL（可选）应参照 RFC 2119 中的说明进行解释。

Implementation-defined（由实现定义）意味着该行为是实现契约的一部分，但本规范并不规定统一的策略。具体实现必须记录所选定的行为

## 1. 问题陈述

Symphony 是一个长期运行的自动化服务，它持续从问题追踪器（本规范版本中为 Linear）读取工作，为每个问题创建一个隔离的工作区，并在工作区内为该问题运行编码代理会话。

该服务解决了四个运营问题：
- 它将问题执行转化为可重复的守护进程工作流，而非手动脚本。
- 它在每个问题的工作区中隔离代理执行，因此代理命令仅在问题工作区目录内运行。
- 它将工作流策略保留在仓库中（`WORKFLOW.md`），以便团队随代码一起对代理提示词和运行时设置进行版本控制。
- 它提供了足够的观察能力来操作和调试多个并发代理运行。

实现者被要求明确记录其信任和安全态势。本规范不要求单一的审批、沙箱或操作员确认策略；某些实现可能针对具有高信任配置的受信环境，而其他实现可能需要更严格的审批或沙箱控制。

重要边界：
- Symphony 是一个调度器/运行器和追踪器读取器。
- 票据写入（状态转换、评论、PR 链接）通常由编码代理使用工作流/运行时环境中的可用工具执行。
- 成功的运行可能结束于工作流定义的移交状态（例如 `Human Review`），而不一定是 `Done`。

## 2. 目标与非目标

### 2.1 目标
- 以固定节奏轮询问题追踪器，并以有限的并发度派发工作。
- 维护唯一的权威编排器状态，用于派发、重试和对账。
- 创建确定性的每个问题的工作区，并在多次运行中保留它们。
- 当问题状态变化使其不符合条件时，停止活动运行。
- 通过指数退避从瞬时失败中恢复。
- 从仓库拥有的 `WORKFLOW.md` 合约中加载运行时行为。
- 暴露操作员可见的观察能力（至少包括结构化日志）。
- 在不要求持久数据库的情况下支持重启恢复。

### 2.2 非目标
- 丰富的 Web UI 或多租户控制平面。
- 规定特定的仪表板或终端 UI 实现。
- 通用工作流引擎或分布式作业调度器。
- 用于编辑票据、PR 或评论的内置业务逻辑（该逻辑存在于工作流提示词和代理工具中）。
- 强制要求超出编码代理和宿主操作系统提供的强大沙箱控制。
- 强制要求所有实现采用单一的默认审批、沙箱或操作员确认态势。

## 3. 系统概览

### 3.1 主要组件
1. `Workflow Loader` (工作流加载器)
   - 读取 `WORKFLOW.md`。
   - 解析 YAML 前导信息（front matter）和提示词正文。
   - 返回 `{config, prompt_template}`。

2. `Config Layer` (配置层)
   - 为工作流配置值暴露类型化读取器。
   - 应用默认值和环境变量间接引用。
   - 在派发前执行编排器使用的验证。

3. `Issue Tracker Client` (问题追踪器客户端)
   - 获取活动状态下的候选问题。
   - 获取特定问题 ID 的当前状态（对账）。
   - 在启动清理期间获取终止状态的问题。
   - 将追踪器负载标准化为稳定的问题模型。

4. `Orchestrator` (编排器)
   - 拥有轮询周期（tick）。
   - 拥有内存中的运行时状态。
   - 决定派发、重试、停止或释放哪些问题。
   - 跟踪会话指标和重试队列状态。

5. `Workspace Manager` (工作区管理器)
   - 将问题标识符映射到工作区路径。
   - 确保存在每个问题的的工作区目录。
   - 运行工作区生命周期钩子。
   - 清理终止问题的工作区。

6. `Agent Runner` (代理运行器)
   - 创建工作区。
   - 根据问题 + 工作流模板构建提示词。
   - 启动编码代理应用服务器客户端。
   - 将代理更新流回传给编排器。

7. `Status Surface` (状态界面，可选)
   - 展示人类可读的运行时状态（例如终端输出、仪表板或其他面向操作员的视图）。

8. `Logging` (日志)
   - 向一个或多个配置的接收器发出结构化运行时日志。

### 3.2 抽象层次
Symphony 在保持这些层级时最容易移植：
1. `Policy Layer` (策略层，仓库定义)
   - `WORKFLOW.md` 提示词正文。
   - 团队特定的票据处理、验证和移交规则。
2. `Configuration Layer` (配置层，类型化读取器)
   - 将前导信息解析为类型化的运行时设置。
   - 处理默认值、环境令牌和路径标准化。
3. `Coordination Layer` (协调层，编排器)
   - 轮询循环、问题资格、并发性、重试、对账。
4. `Execution Layer` (执行层，工作区 + 代理子进程)
   - 文件系统生命周期、工作区准备、编码代理协议。
5. `Integration Layer` (集成层，Linear 适配器)
   - 针对追踪器数据的 API 调用和标准化。
6. `Observability Layer` (可观察性层，日志 + 可选状态界面)
   - 操作员对编排器和代理行为的可见性。

### 3.3 外部依赖
- 问题追踪器 API（本规范版本中针对 `tracker.kind: linear`）。
- 用于工作区和日志的本地文件系统。
- 可选的工作区填充工具（例如 Git CLI，如果使用）。
- 支持通过 stdio 进行类似 JSON-RPC 的应用服务器模式的编码代理可执行文件。
- 用于问题追踪器和编码代理的宿主环境身份验证。

## 4. 核心领域模型

### 4.1 实体
#### 4.1.1 Issue (问题)
由编排、提示词渲染和观察输出使用的标准化问题记录。
字段：
- `id` (字符串): 稳定的追踪器内部 ID。
- `identifier` (字符串): 人类可读的票据编号（例如：`ABC-123`）。
- `title` (字符串)
- `description` (字符串或 null)
- `priority` (整数或 null): 数字越小，派发排序优先级越高。
- `state` (字符串): 当前追踪器状态名称。
- `branch_name` (字符串或 null): 追踪器提供的分支元数据（如果可用）。
- `url` (字符串或 null)
- `labels` (字符串列表): 标准化为小写。
- `blocked_by` (阻碍者引用列表): 每个引用包含 ID、编号和状态。
- `created_at` (时间戳或 null)
- `updated_at` (时间戳或 null)

#### 4.1.2 Workflow Definition (工作流定义)
解析后的 `WORKFLOW.md` 负载：
- `config` (映射): YAML 前导信息根对象。
- `prompt_template` (字符串): 前导信息后修剪过的 Markdown 正文。

#### 4.1.3 Service Config (服务配置，类型化视图)
派生自 `WorkflowDefinition.config` 加环境解析的类型化运行时值。例如：轮询间隔、工作区根目录、活动与终止状态、并发限制、编码代理路径、工作区钩子等。

#### 4.1.4 Workspace (工作区)
分配给一个问题标识符的文件系统工作区。
字段：路径、工作区键（规范化的问题编号）、是否新建标志。

#### 4.1.5 Run Attempt (运行尝试)
针对一个问题的一次执行尝试。
字段：问题 ID、问题编号、尝试次数、工作区路径、开始时间、状态、错误信息。

#### 4.1.6 Live Session (实时会话，代理会话元数据)
编码代理子进程运行时跟踪的状态。
字段：会话 ID、线程 ID、轮次 ID、进程 PID、最近事件、最近消息、令牌消耗统计（Codex 消耗统计）、轮次计数。

#### 4.1.7 Retry Entry (重试条目)
问题的计划重试状态。
字段：问题 ID、编号、尝试次数、到期时间、计时器句柄、错误信息。

#### 4.1.8 Orchestrator Runtime State (编排器运行时状态)
编排器拥有的单一权威内存状态。
字段：轮询间隔、最大并发代理数、运行中映射、已声明集合、重试尝试映射、已完成集合、Codex 总量统计、频率限制快照。

### 4.2 稳定标识符与标准化规则
- `Issue ID`: 用于追踪器查找和内部映射键。
- `Issue Identifier`: 用于人类可读的日志和工作区命名。
- `Workspace Key`: 从 `issue.identifier` 衍生，将非 `[A-Za-z0-9._-]` 字符替换为 `_`。
- `Normalized Issue State`: 转换为小写后比较状态。
- `Session ID`: 组合编码代理的 `thread_id` 和 `turn_id` 为 `<thread_id>-<turn_id>`。

## 5. 工作流规范（仓库合约）

### 5.1 文件发现与路径解析
优先级：显式设置 > 默认（当前工作目录下的 `WORKFLOW.md`）。文件应受版本控制。如果无法读取则报错。

### 5.2 文件格式
带可选 YAML 前导信息的 Markdown 文件。解析规则：以 `---` 开始的部分为 YAML，之后为提示词正文。前导信息必须解析为映射对象。

### 5.3 前导信息架构 (Front Matter Schema)
顶级键：`tracker`, `polling`, `workspace`, `hooks`, `agent`, `codex`。未知键应被忽略以保证向前兼容。扩展键如 `server.port` 可选。

详细字段定义（略，详见原 SPEC 文档关于各对象字段的规定）。

### 5.4 提示词模板合约
`WORKFLOW.md` 的正文是每问题的提示词模板。要求使用严格模板引擎（兼容 Liquid），未知变量/滤镜必须报错。输入变量包括 `issue` 对象和 `attempt` 整数。

### 5.5 验证与错误表面
包括文件缺失、解析错误、模板错误等。文件加载/YAML 错误会阻塞派发；模板错误仅使受影响的运行尝试失败。

## 6. 配置规范
定义了源优先级（规范 > 前导 > 环境 > 默认）、路径解析语义、动态重载语义（需监视文件变更并实时应用）以及派发前的预检验证。

## 7. 编排状态机
编排器是修改状态的唯一组件。定义了问题的内部声明状态（未声明、已声明、运行中、重试排队、已释放）以及运行尝试的生命周期阶段。

## 8. 轮询、调度与对账
定义了轮询循环序列（对账 > 验证 > 获取候选 > 排序 > 派发）。规定了候选选择规则（并发限制、阻碍者规则）、排序规则（优先级 > 创建时间 > 编号）、重试退避公式（Continuation 1秒；失败重试采用指数退避并封顶）以及对账逻辑（停滞检测、追踪器状态同步）。

## 9. 工作区管理与安全
规定了工作区路径布局、创建/复用算法。强调了三项安全不变式：1. 仅在工作区路径运行代理；2. 工作区必须位于根目录内；3. 工作区键必须经过清理净化。

## 10. 代理运行器协议
定义了与编码代理应用服务器集成的语言中性合约。包括启动握手（`initialize`, `initialized`, `thread/start`, `turn/start`）、流式轮次处理（终止条件、续接处理）、运行时事件（`session_started`, `turn_completed` 等）、审批/工具调用政策以及超时/错误映射。

## 11. 问题追踪器集成合约 (Linear 兼容)
要求支持获取候选、按状态获取、按 ID 获取状态的操作。定义了针对 Linear 的 GraphQL 查询语义、标准化规则以及错误处理策略。

## 12. 提示词构建与上下文组装
定义了输入（模板、问题对象、尝试次数）、渲染规则（严格校验）和失败语义。

## 13. 日志、状态与可观察性
规定了日志约定（上下文字段、稳定格式）、 snapshot 接口、令牌/运行时间统计规则、以及人类可读的状态界面规范。
